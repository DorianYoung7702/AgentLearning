# 规划与推理（Planning & Reasoning）

> 规划与推理是 Agent 自主完成复杂任务的核心能力，直接决定 Agent 的"智能程度"。

## 1. 为什么 Agent 需要规划？

简单任务（如翻译一句话）不需要规划，但复杂任务需要：
- **任务分解**：把大任务拆成可执行的小步骤
- **依赖管理**：确定步骤之间的先后顺序
- **资源分配**：选择合适的工具和策略
- **动态调整**：根据中间结果修改后续计划

## 2. 主要规划范式

### 2.1 ReAct（Reasoning + Acting）

**核心思想**：推理和行动交替进行

```
Question: 特斯拉2024年的营收是多少？同比增长了多少？

Thought 1: 我需要先搜索特斯拉2024年的营收数据
Action 1: search("特斯拉 2024年 营收")
Observation 1: 特斯拉2024年营收为977.6亿美元

Thought 2: 我还需要2023年的数据来计算同比增长
Action 2: search("特斯拉 2023年 营收")
Observation 2: 特斯拉2023年营收为967.7亿美元

Thought 3: 现在可以计算同比增长率了
Action 3: calculator("(977.6 - 967.7) / 967.7 * 100")
Observation 3: 1.02%

Thought 4: 我已经有了所有需要的信息
Final Answer: 特斯拉2024年营收为977.6亿美元，同比增长约1.02%。
```

**优缺点**：
- ✅ 灵活，每步可调整方向
- ✅ 推理过程透明可追溯
- ❌ 可能路径不最优，有冗余步骤
- ❌ 长任务中 LLM 可能"忘记"初始目标

### 2.2 Plan-and-Execute

**核心思想**：先规划完整计划，再逐步执行

```
Task: 分析公司Q3销售数据并生成报告

[规划阶段]
Plan:
  1. 从数据库提取Q3销售数据
  2. 按产品线汇总销售额
  3. 与Q2数据对比计算增长率
  4. 识别销售TOP5产品
  5. 生成分析报告

[执行阶段]
Execute Step 1: query_database("SELECT * FROM sales WHERE quarter='Q3'")
→ 结果：获得5000条销售记录

Execute Step 2: analyze_data(data, group_by="product_line")
→ 结果：按产品线汇总完成

... (逐步执行)

[重规划?]
当某步结果与预期不符时，可重新调整后续计划
```

**优缺点**：
- ✅ 执行路径清晰
- ✅ 适合结构化的复杂任务
- ❌ 规划可能不准确，需要重规划机制
- ❌ 前期规划消耗额外 Token

### 2.3 ReWOO（Reasoning WithOut Observation）

**核心思想**：先一次性规划所有步骤和工具调用，再批量执行

```
Plan:
  #E1 = search("特斯拉 2024营收")
  #E2 = search("特斯拉 2023营收")
  #E3 = calculator(#E1的营收 - #E2的营收) / #E2的营收 * 100

Execute: 并行执行 #E1 和 #E2，再执行 #E3

Solve: 基于所有结果生成最终答案
```

**优缺点**：
- ✅ 减少 LLM 调用次数（只规划一次）
- ✅ 可并行执行独立步骤
- ❌ 无法根据中间结果调整计划

## 3. 推理策略

### 3.1 Chain-of-Thought（CoT）

线性逐步推理（详见 Prompt Engineering 章节）

### 3.2 Tree of Thoughts（ToT）

```
核心流程：
1. 对问题生成多个候选"思路"
2. 对每个思路进行评估打分
3. 选择最优思路继续展开
4. 可以回溯到之前的节点尝试其他路径

搜索策略：
- BFS（广度优先）：适合解空间较小的问题
- DFS（深度优先）：适合解空间大但有明确评估标准的问题
```

### 3.3 Graph of Thoughts（GoT）

将思维过程组织为**图结构**，允许：
- 思路之间相互引用和合并
- 比 ToT 更灵活的拓扑结构
- 支持思路的聚合操作

### 3.4 Self-Consistency

```
对同一问题进行 N 次 CoT 推理（temperature > 0）
→ 得到 N 条推理路径和 N 个答案
→ 多数投票选出最终答案

示例：
路径1 → 答案A
路径2 → 答案A  
路径3 → 答案B
路径4 → 答案A
→ 最终答案：A（3/4 投票）
```

### 3.5 Reflexion（反思）

```
循环：
  1. 执行任务 → 得到结果
  2. 评估结果 → 是否成功？
  3. 如果失败：
     - 分析失败原因
     - 生成改进建议（存入记忆）
     - 基于反思经验重新执行
  4. 如果成功：结束
```

## 4. 任务分解策略

### 4.1 自顶向下分解

```
大任务
├── 子任务1
│   ├── 子任务1.1
│   └── 子任务1.2
├── 子任务2
└── 子任务3
    ├── 子任务3.1
    └── 子任务3.2
```

### 4.2 分解原则

1. **MECE 原则**：相互独立、完全穷尽
2. **粒度适中**：每个子任务对应一次工具调用或简单推理
3. **依赖明确**：标注哪些子任务可以并行
4. **可验证**：每个子任务有明确的完成标准

### 4.3 常见分解模式

| 模式 | 说明 | 示例 |
|------|------|------|
| 顺序分解 | 步骤按序执行 | 数据采集 → 清洗 → 分析 → 报告 |
| 并行分解 | 独立子任务并行 | 同时搜索多个来源 |
| 条件分解 | 基于结果分支 | if 有数据 → 分析, else → 采集 |
| 迭代分解 | 重复执行直到满足条件 | 迭代优化文案直到满意 |

## 5. 规划的挑战与应对

### 5.1 规划失败的常见原因

| 原因 | 说明 | 应对 |
|------|------|------|
| 目标理解偏差 | LLM 误解了用户意图 | 明确任务定义，支持澄清 |
| 分解不合理 | 子任务粒度或逻辑不对 | 提供分解示例，限制分解层级 |
| 忽略依赖 | 未考虑步骤间的顺序关系 | 显式标注依赖关系 |
| 过度规划 | 规划太细导致僵化 | 采用渐进式规划 |
| 无法重规划 | 中间失败后不会调整 | 增加重规划触发机制 |

### 5.2 重规划（Re-planning）机制

```
触发条件：
- 工具调用返回了意外结果
- 某个子任务执行失败
- 用户中途修改了需求
- 发现新信息改变了问题性质

重规划流程：
1. 评估当前进度和已完成的步骤
2. 分析失败/变化的原因
3. 基于新信息重新规划后续步骤
4. 继续执行
```

## 6. 规划范式对比总结

| 范式 | 规划时机 | 灵活性 | Token 效率 | 适用场景 |
|------|---------|--------|-----------|---------|
| ReAct | 每步规划 | 最高 | 低（多次LLM调用） | 探索性任务 |
| Plan-Execute | 先规划后执行 | 中 | 中 | 结构化任务 |
| ReWOO | 一次规划 | 低 | 最高 | 简单多步任务 |
| ToT | 每步多路探索 | 高 | 最低 | 推理密集型 |
| Reflexion | 执行后反思 | 高 | 低 | 试错型任务 |

## 7. 面试高频问题

### Q1: 对比 ReAct 和 Plan-and-Execute 两种规划范式。
**要点**：ReAct 逐步规划+执行，灵活但可能冗余；Plan-Execute 先全局规划再执行，路径清晰但需要重规划机制。选择取决于任务结构化程度。

### Q2: Agent 规划失败时应该怎么处理？
**要点**：① 检测失败（工具返回异常/结果不符预期）② 分析原因 ③ 重规划后续步骤 ④ 设置最大重试次数 ⑤ 必要时降级或请求用户帮助。

### Q3: 如何将一个复杂任务合理分解为子任务？
**要点**：遵循 MECE 原则，粒度适中（每个子任务对应一次工具调用），明确标注依赖关系和可并行的部分，每个子任务要有可验证的完成标准。

### Q4: Tree of Thoughts 和 Chain of Thought 有什么区别？
**要点**：CoT 是线性单路径推理；ToT 是树形多路径探索，每步生成多个候选并评估，可回溯。ToT 更适合需要创造性或复杂推理的任务，但成本更高。

### Q5: 什么是 Reflexion？它在什么场景下有用？
**要点**：Reflexion 让 Agent 执行后自我反思，将失败经验作为未来行动的指导。适用于可重试的任务（如编程调试、写作迭代），通过积累经验逐步改进。
