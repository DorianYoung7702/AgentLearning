# Agent 评估与基准测试

> Agent 的质量保障是工程落地的关键，面试中评估能力是区分"会用"和"能落地"的分水岭。

## 1. 为什么 Agent 评估很难？

| 挑战 | 说明 |
|------|------|
| **非确定性** | 同一输入可能产生不同的执行路径和结果 |
| **多步骤** | 需要评估每一步和最终结果 |
| **开放性** | 许多任务没有唯一正确答案 |
| **工具交互** | 需要模拟或真实执行工具调用 |
| **上下文依赖** | 评估结果受上下文和历史影响 |

## 2. 评估维度

### 2.1 核心评估维度

```
Agent 评估
├── 任务完成度（Task Completion）
│   ├── 最终结果正确性
│   └── 部分完成度
├── 效率（Efficiency）
│   ├── 步骤数
│   ├── Token 消耗
│   └── 延迟
├── 可靠性（Reliability）
│   ├── 成功率
│   ├── 错误恢复能力
│   └── 一致性
├── 安全性（Safety）
│   ├── 幻觉率
│   ├── 有害输出
│   └── 权限遵守
└── 用户体验（UX）
    ├── 响应质量
    ├── 对话流畅度
    └── 透明度
```

### 2.2 具体指标

| 指标 | 计算方式 | 说明 |
|------|---------|------|
| **成功率** | 成功任务数 / 总任务数 | 最基础的指标 |
| **平均步骤数** | 总步骤 / 任务数 | 衡量效率 |
| **工具调用准确率** | 正确调用数 / 总调用数 | 衡量工具使用能力 |
| **幻觉率** | 含幻觉的回答 / 总回答 | 衡量可靠性 |
| **Token 成本** | 输入 + 输出 Token 数 | 衡量成本 |
| **首字延迟（TTFT）** | 首个 Token 的响应时间 | 用户体验 |
| **端到端延迟** | 任务完成总时间 | 效率 |

## 3. 评估方法

### 3.1 离线评估（Offline Evaluation）

```
测试集准备：
  构建 (输入, 期望输出, 评估标准) 三元组

执行方式：
  遍历测试集 → Agent 执行 → 对比输出与期望 → 计算指标

适用场景：
  开发阶段、回归测试、框架对比
```

### 3.2 在线评估（Online Evaluation）

```
生产环境中的持续监控：
  - 用户满意度评分
  - 任务完成率追踪
  - 错误率和异常监控
  - A/B 测试不同策略
```

### 3.3 LLM-as-Judge（LLM 评估 LLM）

用另一个 LLM 来评判 Agent 的输出质量：

```
Prompt：
请评估以下 Agent 的回答质量，从 1-5 打分。

评估维度：
- 准确性：回答是否事实正确
- 相关性：是否切题
- 完整性：是否覆盖要点
- 清晰度：表达是否清楚

Agent 的问题：{question}
Agent 的回答：{answer}
参考答案：{reference}（可选）

请给出分数和理由。
```

**优缺点**：
- ✅ 可扩展，处理开放式任务
- ✅ 成本低于人工评估
- ❌ 评估本身可能有偏差
- ❌ 需要校准评估标准

### 3.4 人工评估

- **众包评估**：大量评估者打分
- **专家评估**：领域专家深度评估
- **对比评估**：A/B 两个版本人工比较

## 4. 主流评估基准

### 4.1 通用 Agent 基准

| 基准 | 说明 | 考察能力 |
|------|------|---------|
| **AgentBench** | 综合 Agent 评估 | 代码、知识、Web、游戏等 |

## 5. 构建评估体系

### 5.1 评估流水线

```
1. 定义评估场景和用例
   ↓
2. 构建测试数据集
   ├── 正常用例
   ├── 边缘用例
   └── 对抗用例
   ↓
3. 选择评估指标
   ├── 自动指标
   └── 人工指标
   ↓
4. 执行评估
   ├── 离线评估
   └── 在线监控
   ↓
5. 分析结果
   ├── 错误分类
   └── 改进方向
   ↓
6. 迭代优化 → 回到步骤4
```

### 5.2 测试数据集构建

**分层构建**：
```
Level 1: 基础能力测试
  - 单工具调用
  - 简单问答
  - 格式正确性

Level 2: 组合能力测试
  - 多步推理
  - 多工具组合
  - 上下文理解

Level 3: 复杂场景测试
  - 长对话保持
  - 错误恢复
  - 边缘情况

Level 4: 对抗测试
  - 提示注入（Prompt Injection）
  - 越界请求
  - 矛盾信息
```

### 5.3 回归测试

```
每次修改（Prompt/模型/工具/流程）后：
  1. 运行完整测试集
  2. 对比修改前后指标
  3. 发现指标下降 → 回退或修复
  4. 所有指标不退化 → 发布
```

## 6. 常见评估陷阱

| 陷阱 | 说明 | 避免方法 |
|------|------|---------|
| 数据泄露 | 测试数据在训练集中 | 定期更新测试集 |
| 过度拟合测试集 | 针对测试集优化 | 保留 holdout 集 |
| 忽视分布外数据 | 只测试典型场景 | 包含对抗和边缘用例 |
| 只看平均分 | 隐藏了尾部问题 | 分析分位数和失败模式 |
| 评估不一致 | 同一输出不同时间评分不同 | 标准化评估流程 |

## 7. 面试高频问题

### Q1: 如何评估一个 Agent 系统的质量？
**要点**：从多维度评估 — 任务完成度、效率（步骤数/Token/延迟）、可靠性（成功率/一致性）、安全性（幻觉率）。结合离线测试集 + 在线监控 + LLM-as-Judge。

### Q2: 什么是 LLM-as-Judge？有什么局限性？
**要点**：用 LLM 评估另一个 LLM 的输出，通过提供评估标准和维度让 LLM 打分。局限性：位置偏差（倾向选第一个）、冗长偏差（倾向选更长的）、自我偏差。可通过交换位置、多次评估缓解。

### Q3: 如何构建 Agent 的测试数据集？
**要点**：分层构建 — 基础能力 → 组合能力 → 复杂场景 → 对抗测试。覆盖正常用例和边缘情况。定期更新防止过拟合。人工标注 + LLM 辅助生成。

### Q4: Agent 上线后如何做持续监控？
**要点**：① 关键指标看板（成功率、延迟、成本）② 错误分类和告警 ③ 用户反馈收集 ④ 定期回放分析 ⑤ A/B 测试新版本。

### Q5: 如何测试 Agent 的安全性？
**要点**：① Prompt Injection 攻击测试 ② 越界请求测试 ③ 敏感信息泄露测试 ④ 工具滥用测试 ⑤ 红队测试（Red Teaming）。
