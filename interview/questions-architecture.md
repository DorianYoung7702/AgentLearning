## 一、可观测性与运维

### A1: 如何设计 Agent 系统的全链路追踪？

**参考要点**：
- 基于 Trace-Span 模型：一次交互 = 一个 Trace，每个步骤 = 一个 Span
- 每个 Span 记录：类型（LLM/Tool/Retrieval）、输入输出、Token 数、延迟、成本
- 支持 Span 嵌套：如 LLM 调用内部的 Embedding 调用
- 工具选型：LangFuse（开源可自建）/ LangSmith（LangChain 生态）
- 与现有 APM 集成：导出到 OpenTelemetry / Prometheus

### A2: Agent 上线后发现成功率从 90% 掉到 70%，你怎么排查？

**参考要点**：
1. **看监控**：哪些指标异常？是全部请求还是特定类型？
2. **分层定位**：
   - LLM API 层：模型服务是否异常？延迟是否飙升？
   - 工具层：某个工具是否故障？失败率？
   - 数据层：知识库是否更新导致检索质量下降？
   - Prompt 层：是否有 Prompt 变更？
3. **回放分析**：对比失败和成功的 Trace，找差异
4. **分群分析**：按用户/场景/工具分组，定位影响范围
5. **快速止血**：回滚最近变更 / 切换备用模型

### A3: 如何做 Agent 的灰度发布？

**参考要点**：
- **流量维度**：按用户 ID / 请求比例分流
- **变更维度**：Prompt 变更、模型变更、工具变更、流程变更
- **观察指标**：成功率、延迟、成本、用户反馈
- **灰度流程**：1% -> 5% -> 20% -> 50% -> 100%，每步观察 30 分钟
- **自动回滚**：核心指标退化超过阈值自动回滚
- **实现方式**：Feature Flag + A/B 测试框架

## 二、系统架构设计

### A4: 设计一个企业级 Agent 平台（支持多租户）

**参考要点**：

**需求**：多个业务方可创建和管理自己的 Agent，互相隔离。

**架构**：
```
租户管理层：
  - 租户认证鉴权
  - 资源配额管理（Token 上限、并发数）
  - 计费系统

Agent 管理层：
  - Prompt 版本管理
  - 工具注册和管理
  - 知识库管理（向量库隔离）

执行层：
  - Agent 编排引擎
  - LLM Gateway（多模型路由 + 限流）
  - 工具执行沙箱

基础设施层：
  - 向量数据库（Collection 级租户隔离）
  - 状态存储（Redis / PostgreSQL）
  - 消息队列（异步任务）
  - 监控和日志
```

**关键设计决策**：
- 隔离级别：逻辑隔离（共享集群 + 命名空间）vs 物理隔离
- 模型选择：租户可选不同模型，平台统一网关
- 安全：租户之间数据严格隔离，Prompt 防泄露

### A5: 设计一个 LLM Gateway

**参考要点**：

**核心功能**：
1. **路由**：按任务类型/复杂度选择模型
2. **负载均衡**：多 API Key 轮询，多提供商分流
3. **限流**：全局 RPM/TPM + 租户级配额
4. **降级**：主模型超时/失败 -> 备用模型 -> 降级响应
5. **缓存**：语义缓存减少重复调用
6. **统计**：Token/延迟/成本/错误率实时统计

**架构决策**：
- 代理模式（Proxy）vs SDK 模式
- 同步 vs 异步
- 缓存粒度（完全匹配 vs 语义相似度）

### A6: 设计一个支持长时间运行的异步 Agent 系统

**参考要点**：

**场景**：数据分析 Agent，任务可能运行 5-30 分钟。

**架构**：
```
用户 -> API (返回 task_id) -> 消息队列 (Kafka)
  -> Agent Worker (消费执行)
  -> State Store (Redis + PG)
  -> 用户查询进度 (WebSocket/轮询)
  -> 完成后通知 (WebSocket/回调/邮件)
```

**关键设计**：
- Checkpoint 机制：每步持久化状态，支持断点恢复
- Worker 弹性伸缩：根据队列深度自动扩缩容
- 超时处理：全局超时 + 单步超时
- 优先级队列：VIP 用户优先处理
- 结果存储：大结果（报告/文件）存 S3，元数据存 PG

## 三、工程实践

### A7: 如何做 Agent 的 Prompt 管理？

**参考要点**：
- **版本控制**：Git 管理，PR Review
- **模块化**：角色/工具/规则/格式/示例分文件维护
- **A/B 测试**：新版本灰度对比
- **评估基准**：变更前后自动对比核心指标
- **回滚机制**：指标退化快速回滚
- **环境隔离**：dev / staging / production 不同 Prompt

### A8: 如何设计 Agent 的错误处理和降级策略？

**参考要点**：

```
错误分类和处理：

LLM API 错误：
  - 429 限流 -> 指数退避重试 -> 切换 API Key -> 切换模型
  - 500 服务错误 -> 重试 2 次 -> 切换备用模型
  - 超时 -> 重试 1 次 -> 返回部分结果

工具调用错误：
  - 参数错误 -> 让 LLM 修正重试 (最多 2 次)
  - 工具不可用 -> 跳过工具，直接回答
  - 结果为空 -> 换查询方式重试

Agent 逻辑错误：
  - 输出格式错误 -> JSON 容错解析
  - 死循环 -> 最大步骤限制 (如 10 步)
  - 超出 Token 预算 -> 强制终止，返回已有结果

降级策略（从高到低）：
  1. 完整 Agent (全功能)
  2. 简化 Agent (减少推理步骤)
  3. RAG 直接检索 (不做推理)
  4. 模板化回答 (预定义回复)
  5. 转人工
```

### A9: Agent 系统的测试策略应该如何设计？

**参考要点**：
- 四层测试：Prompt Lint -> 单元测试 -> 集成测试(Mock) -> E2E
- Mock LLM：预定义 LLM 响应，测试 Agent 流程逻辑
- 测试数据集：Golden Set + Regression Set + Edge Case + Adversarial
- 回归测试：CI/CD 集成，变更后自动运行
- 评估指标：成功率、步骤数、Token 消耗、格式正确率
- 性能测试：并发能力、延迟分布

### A10: 如何设计语义缓存提升 Agent 性能？

**参考要点**：
- **原理**：相似的 Query 返回缓存的 Response，省去 LLM 调用
- **实现**：Query 向量化 -> 在缓存向量库中搜索 -> 相似度 > 0.95 命中
- **缓存策略**：TTL（根据数据时效）+ LRU（控制缓存大小）
- **失效机制**：知识库更新时清除相关缓存
- **适用场景**：FAQ 类查询命中率高；创意/个性化查询不适合缓存
- **注意**：缓存粒度、多轮对话上下文影响、缓存命中率监控

## 四、技术选型与决策

### A11: 技术选型：自研 Agent 框架 vs 使用开源框架？

**参考要点**：

| 维度 | 自研 | 开源框架 |
|------|------|---------|
| 上手速度 | 慢 | 快 |
| 定制灵活性 | 最高 | 受框架限制 |
| 维护成本 | 高（自己维护） | 中（依赖社区） |
| 调试难度 | 低（代码全掌控） | 高（封装层多） |
| 人才招聘 | 需要内部培训 | 社区经验可用 |

**决策框架**：
- 小团队 / 快速验证 -> 用框架
- 核心业务 / 有定制需求 -> 框架 + 部分自研
- 平台级产品 / 大团队 -> 自研核心 + 参考框架设计

### A12: 模型选型：云 API vs 私有化部署？

**参考要点**：

| 维度 | 云 API | 私有化部署 |
|------|--------|-----------|
| 启动成本 | 低（按量付费） | 高（GPU 采购） |
| 数据安全 | 数据出境风险 | 数据不出内网 |
| 延迟 | 网络延迟 | 内网低延迟 |
| 灵活性 | 受限于 API | 可微调定制 |
| 运维 | 厂商负责 | 自己运维 |
| 长期成本 | 按量计费，可能更贵 | 固定投入，量大更划算 |

**决策因素**：数据合规要求、调用量级、定制需求、团队运维能力。

### A13: 如何做模型分级路由？

**参考要点**：
```
路由策略：

1. 基于规则的路由
   - 意图分类 -> 简单查询用 mini，复杂推理用 4o
   - 输入长度 -> 短输入用快速模型

2. 基于分类模型的路由
   - 训练一个轻量分类器判断任务复杂度
   - 低/中/高 -> 对应不同模型

3. 级联路由
   - 先用小模型尝试 -> 自评估置信度
   - 置信度低 -> 升级到大模型

实际效果：
  - 70% 请求走 mini 模型 (成本低 20x)
  - 25% 请求走标准模型
  - 5% 复杂请求走旗舰模型
  - 综合节省 60-70% 成本
```

## 五、综合场景题

### A14: 你负责一个 Agent 项目，从 0 到 1 的技术规划是什么？

**参考要点**（按阶段回答）：

**Phase 1 - 验证（2 周）**
- 确定核心场景和 MVP 功能
- 直接用 OpenAI API 快速搭建原型
- 收集 50 个测试用例验证效果

**Phase 2 - 框架搭建（4 周）**
- 搭建 Agent 基础架构（编排引擎/工具管理/记忆系统）
- 接入向量数据库，构建 RAG 管道
- 建立测试体系（Golden Set + CI/CD）
- 接入可观测性工具

**Phase 3 - 优化上线（4 周）**
- Prompt 优化迭代
- 成本优化（模型分级/缓存）
- 安全加固（Guardrails/Prompt Injection 防御）
- 灰度发布 -> 全量上线
- 建立监控告警

**Phase 4 - 持续迭代**
- 基于用户反馈持续改进
- 扩展工具和场景
- 探索新技术（MCP、推理模型等）

### A15: 你的 Agent 系统面对 10 倍流量增长，需要做哪些调整？

**参考要点**：
1. **LLM 层**：多 API Key、多模型提供商、语义缓存、模型分级降低成本
2. **计算层**：Agent Service 水平扩容、异步队列削峰
3. **存储层**：向量数据库分片、Redis 集群、数据库读写分离
4. **成本层**：Token 预算限制、请求频率限制、模型降级策略
5. **监控层**：告警阈值调整、容量预警
6. **架构层**：考虑从同步转异步、引入消息队列

## 答题模板

架构题建议用 **STAR + 技术深度** 模式回答：

```
1. 场景分析 (30s)
   - 核心需求是什么
   - 约束条件（性能/成本/安全/规模）

2. 方案设计 (2-3min)
   - 画架构图（即使是口述也描述清楚组件和数据流）
   - 核心组件和职责

3. 技术选型 (1min)
   - 每个选择说清楚 why

4. 关键挑战 (1-2min)
   - 主动说出潜在问题和解决方案
   - 这是体现深度的地方

5. 演进思路 (30s)
   - 从 MVP 到生产的路径
   - 未来可能的扩展
```
