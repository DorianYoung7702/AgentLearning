# 大语言模型（LLM）基础

> Agent 的核心是 LLM，理解 LLM 的原理和局限性是构建可靠 Agent 的基础。
> **最后更新：2026 年 4 月**

## 1. Transformer 架构

### 1.1 核心机制：自注意力（Self-Attention）

Transformer 的核心是**自注意力机制**，它让模型在处理一个 Token 时，能"关注"输入序列中所有其他 Token。

**计算过程**：
```
输入: X (序列中每个 Token 的 Embedding)

1. 线性投影生成 Q, K, V:
   Q = X·W_Q,  K = X·W_K,  V = X·W_V

2. 计算注意力权重:
   Attention(Q, K, V) = softmax(Q·K^T / √d_k) · V

其中 √d_k 是缩放因子，防止点积值过大
```

**多头注意力（Multi-Head Attention）**：
- 将 Q、K、V 分成多个"头"，每个头独立计算注意力
- 不同的头可以关注不同类型的信息（语法、语义、位置等）
- 最后拼接所有头的输出

### 1.2 位置编码（Positional Encoding）

Transformer 本身对位置不敏感，需要加入位置信息：
- **绝对位置编码**：正弦/余弦函数（原始 Transformer）
- **旋转位置编码（RoPE）**：LLaMA 系列使用，支持外推更长序列
- **ALiBi**：通过注意力偏置编码位置，无需额外参数

### 1.3 Decoder-Only 架构

现代 LLM（GPT、LLaMA、Qwen 等）普遍采用 Decoder-Only 架构：
- 使用**因果注意力掩码（Causal Mask）**：每个 Token 只能关注自己和之前的 Token
- 自回归生成：逐 Token 生成，每次将之前生成的 Token 作为输入

## 2. LLM 的训练流程

### 三阶段训练

```
预训练 (Pre-training)
    │ 海量文本无监督学习，学习语言知识
    ▼
监督微调 (SFT - Supervised Fine-Tuning)
    │ 用高质量指令-回答对进行有监督训练
    ▼
人类偏好对齐 (RLHF / DPO)
    │ 让模型输出更符合人类偏好
    ▼
最终模型
```

### 2.1 预训练
- **目标**：Next Token Prediction（预测下一个 Token）
- **数据**：互联网文本、书籍、代码等，通常数万亿 Token
- **结果**：模型获得语言理解和世界知识

### 2.2 监督微调（SFT）
- **目标**：让模型学会遵循指令
- **数据**：(指令, 回答) 对，通常数万到数百万条
- **结果**：模型变成可用的"助手"

### 2.3 人类偏好对齐
- **RLHF（基于人类反馈的强化学习）**：
  - 训练一个奖励模型（Reward Model）评分
  - 使用 PPO 等算法优化策略
- **DPO（直接偏好优化）**：
  - 不需要单独的奖励模型
  - 直接用偏好对（chosen vs rejected）训练

## 3. LLM 推理

### 3.1 生成策略

| 参数 | 说明 | 对 Agent 的影响 |
|------|------|----------------|
| **Temperature** | 控制输出随机性，0=确定性，1=更随机 | Agent 通常用较低温度（0-0.3）保证可靠性 |
| **Top-p** | 核采样，从累积概率前 p 的 Token 中采样 | 常设 0.9-0.95 |
| **Top-k** | 从概率最高的 k 个 Token 中采样 | 限制采样范围 |
| **Max Tokens** | 最大生成 Token 数 | 需根据任务复杂度设置 |
| **Stop Sequences** | 停止生成的标记 | Agent 中用于识别工具调用边界 |

### 3.2 上下文窗口（Context Window）

- **定义**：模型一次能处理的最大 Token 数
- **常见大小**：4K → 8K → 32K → 128K → **1M**（2026 旗舰模型标配）
- **对 Agent 的影响**：
  - 决定了 Agent 能"记住"多少历史对话和工具结果
  - 上下文过长会降低模型注意力质量（"Lost in the Middle" 问题）
  - 需要合理管理上下文，避免超出窗口限制

### 3.3 KV Cache

- LLM 推理时缓存已计算的 Key 和 Value，避免重复计算
- **对 Agent 的影响**：多轮对话中，KV Cache 显著加速推理
- **优化技术**：GQA（分组查询注意力）、MQA（多查询注意力）减少 KV Cache 大小

## 4. Token 与 Tokenization

### 4.1 Tokenizer

将文本切分为 Token（模型处理的最小单位）：
- **BPE（Byte-Pair Encoding）**：GPT 系列使用
- **SentencePiece**：LLaMA 系列使用
- **中文**通常 1 个汉字 ≈ 1-2 个 Token

### 4.2 Token 计费

- API 调用按 Token 数计费（输入 + 输出）
- Agent 的多步执行会累积大量 Token 消耗
- 成本优化是 Agent 工程中的重要考量

### 选型考量因素
1. **能力匹配**：任务对推理/代码/多模态的需求

6. **合规**：数据是否能出境，是否需要私有化部署

## 6. LLM 的局限性（面试重点）

| 局限性 | 说明 | Agent 中的应对策略 |
|--------|------|-------------------|
| **幻觉** | 生成看似合理但错误的内容 | RAG 引入外部知识、工具验证 |
| **知识截止** | 训练数据有截止日期 | 搜索工具获取最新信息 |
| **推理不稳定** | 复杂推理可能出错 | CoT、多步验证、Self-consistency |
| **上下文有限** | Token 窗口限制 | 摘要压缩、记忆管理 |
| **无法行动** | LLM 本身只能生成文本 | 工具调用弥补 |

## 7. 面试高频问题

### Q1: 解释 Transformer 中自注意力机制的工作原理？
**要点**：Q/K/V 三个矩阵，Q·K^T 计算相关性得分，softmax 归一化，乘以 V 得到加权输出。缩放因子√d_k 防止梯度消失。

### Q2: Temperature 参数在 Agent 中应该怎么设置？为什么？
**要点**：Agent 场景通常用低温度（0-0.3），因为需要可靠、一致的输出（工具调用格式、JSON 解析等）。创意类任务可适当提高。

### Q3: 如何解决 LLM 的幻觉问题？
**要点**：① RAG 引入事实来源 ② 工具调用验证信息 ③ Self-consistency（多次生成取一致结果）④ 引用来源要求 ⑤ 置信度判断

### Q4: 对比 RLHF 和 DPO 的优缺点。
**要点**：RLHF 需要训练奖励模型+PPO，训练复杂但效果经过验证；DPO 直接用偏好对训练，实现简单但可能不如 RLHF 在复杂场景的效果。

### Q5: 为什么现代 LLM 大多采用 Decoder-Only 架构？
**要点**：① 自回归生成自然适合文本生成 ② 训练目标简单统一 ③ 扩展性好（scaling law）④ 通过 In-Context Learning 可处理各种任务，无需 Encoder。
