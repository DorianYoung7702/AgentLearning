# 进阶面试题集

> 涵盖 RAG、多Agent、框架选型、评估、部署等进阶话题。

---

## 一、RAG（检索增强生成）

### Q1: 描述 RAG 的完整流程，包括离线和在线阶段。

**参考要点**：
- **离线**：文档加载 → 文本切分（Chunking）→ 向量化（Embedding）→ 存入向量DB
- **在线**：查询处理（改写/扩展）→ 检索（向量+关键词）→ Rerank → 上下文组装 → LLM 生成回答

### Q2: RAG vs Fine-tuning，各自适用什么场景？

**参考要点**：
| 维度 | RAG | Fine-tuning |
|------|-----|------------|
| 知识更新 | 实时更新知识库即可 | 需要重新训练 |
| 可追溯性 | 可引用来源文档 | 无法追溯 |
| 数据量要求 | 少量文档即可 | 需要大量训练数据 |
| 适用场景 | 知识问答、文档查询 | 风格/格式学习、推理模式 |
| 可组合 | 两者可结合使用 | |

### Q3: 如何选择 Chunk 大小？影响因素有哪些？

**参考要点**：
- 通常 256-512 Token，overlap 50-100 Token
- 小 Chunk（100-200）：检索精准但缺上下文
- 大 Chunk（500-1000）：上下文丰富但有噪声
- 影响因素：文档类型、查询粒度、Embedding 模型性能、上下文窗口大小
- 最佳实践：实验对比，用评估指标选择最优

### Q4: 什么是混合检索？为什么比单一检索好？

**参考要点**：
- 混合检索 = 向量检索（语义）+ 关键词检索（精确）
- 向量弥补关键词的语义不足（"汽车" vs "轿车"）
- 关键词弥补向量的精确性不足（特定术语、编号）
- 融合方法：RRF（Reciprocal Rank Fusion）

### Q5: 什么是 Rerank？为什么需要它？

**参考要点**：
- 初始检索是粗排（效率优先），可能不够精准
- Reranker 用更强的模型（Cross-Encoder）对候选集精细评分
- 流程：检索 Top-50 → Reranker 重新打分 → 取 Top-5
- 在混合检索后效果提升尤其明显

### Q6: 什么是 HyDE？它解决了什么问题？

**参考要点**：
- HyDE = Hypothetical Document Embeddings
- 让 LLM 先假设一个答案文档，用这个假设文档的 Embedding 做检索
- 解决 Query 和 Document 之间的语义鸿沟（用户的简短提问 vs 详细的文档内容）
- 假设文档的 Embedding 更接近真实文档

### Q7: 如何评估 RAG 系统的效果？

**参考要点**：
- 检索质量：Recall@K、MRR、NDCG
- 生成质量：Faithfulness（忠实度）、Relevance（相关性）、Completeness（完整性）
- 使用 RAGAS 等框架自动化评估
- 重点关注 Faithfulness（不产生幻觉）

### Q8: 描述 Self-RAG 和 Corrective RAG 的思路。

**参考要点**：
- **Self-RAG**：Agent 自主判断是否需要检索 → 检索后评估结果是否有用 → 决定是否使用
- **CRAG**：检索后评估文档质量 → 高质量直接用 / 模糊则补充检索 / 低质量丢弃换 Web 搜索
- 两者都是对 Naive RAG 的改进，增加了检索决策和质量评估环节

## 二、规划与推理

### Q9: 对比 ReAct、Plan-and-Execute、ReWOO 三种规划范式。

**参考要点**：
| | ReAct | Plan-Execute | ReWOO |
|---|-------|-------------|-------|
| 规划时机 | 每步规划 | 先整体规划 | 一次性规划 |
| 灵活性 | 最高 | 中 | 最低 |
| Token 效率 | 低 | 中 | 最高 |
| 适用场景 | 探索性任务 | 结构化任务 | 简单多步任务 |

### Q10: 什么是 Tree of Thoughts（ToT）？

**参考要点**：
- 将推理组织为树形结构，每步生成多个候选思路
- 对每个候选评估打分，选择最优路径继续
- 支持回溯到之前的节点尝试其他路径
- 搜索策略：BFS（小解空间）/ DFS（大解空间）
- 适合需要创造性或复杂推理的任务，但成本高

### Q11: 什么是 Reflexion？在什么场景下有用？

**参考要点**：
- 执行后自我反思，将失败经验存入记忆指导后续行动
- 循环：执行 → 评估 → 失败则反思 → 基于经验重试
- 适用场景：编程调试、写作迭代、策略优化等可重试任务

### Q12: Agent 规划失败时如何处理？设计一个重规划机制。

**参考要点**：
- 触发条件：工具返回异常 / 子任务失败 / 用户修改需求 / 发现新信息
- 处理流程：评估当前进度 → 分析失败原因 → 重新规划后续步骤 → 继续执行
- 设置最大重规划次数
- 必要时降级或请求用户帮助

## 三、多 Agent

### Q13: 多 Agent 系统有哪些架构模式？

**参考要点**：
- **中心化（Orchestrator）**：编排器分配任务，控制力强但单点瓶颈
- **去中心化（P2P）**：Agent 直接通信，灵活但协调复杂
- **层级式**：多层管理，适合大型项目
- **流水线**：Agent 按序处理，适合阶段性任务

### Q14: 多 Agent 之间如何通信？

**参考要点**：
1. **消息传递**：异步发送结构化消息
2. **共享状态**：全局状态对象，Agent 读写同一数据
3. **对话/辩论**：Agent 之间通过对话交换观点
- 选择取决于协作模式和实时性要求

### Q15: 如何避免多 Agent 的无限循环？

**参考要点**：
- 设置全局最大轮次和 Token 预算
- 超时机制（单步超时 + 全局超时）
- 检测重复消息并中断
- Orchestrator 有权强制终止
- 明确定义任务完成条件

### Q16: 什么时候该用多 Agent，什么时候单 Agent 就够？

**参考要点**：
- 单 Agent：任务简单、同质、不需要多种专业知识
- 多 Agent：需要多种专业知识、可并行分解、需要交叉验证
- 注意多 Agent 的额外通信成本和复杂性
- 不要为了多 Agent 而多 Agent

## 四、框架选型

### Q17: 对比 LangChain 和 LlamaIndex。

**参考要点**：
| | LangChain | LlamaIndex |
|---|-----------|------------|
| 侧重 | 通用 Agent 框架 | 数据/RAG 框架 |
| Agent 能力 | 强 | 中 |
| RAG 能力 | 中 | 强 |
| 数据连接器 | 少 | 多（160+）|
| 选择依据 | 需要复杂 Agent 工作流 | 以 RAG 为主 |

### Q18: LangGraph 解决了什么问题？

**参考要点**：
- 用有向图定义 Agent 工作流
- 解决了传统 Chain 无法表达循环和条件分支的问题
- 支持持久化状态（checkpoint）和 Human-in-the-loop
- 核心概念：State、Node、Edge、Graph

### Q19: 使用框架 vs 不使用框架，如何选择？

**参考要点**：
- **不用框架**：简单场景、需要极致性能、不想引入依赖
- **用框架**：快速原型、需要丰富集成、团队协作
- 缺点：额外抽象、版本不稳定、调试困难、性能开销

## 五、评估与部署

### Q20: 如何评估一个 Agent 系统的质量？

**参考要点**：
- 多维度评估：任务完成度、效率、可靠性、安全性
- 方法：离线测试集 + 在线监控 + LLM-as-Judge + 人工评估
- 关键指标：成功率、步骤数、Token 消耗、幻觉率、延迟

### Q21: 什么是 LLM-as-Judge？有什么局限性？

**参考要点**：
- 用 LLM 评估另一个 LLM 的输出质量
- 局限性：位置偏差、冗长偏差、自我偏差
- 缓解：交换位置评估、多次评估取平均、不同模型交叉评估

### Q22: 如何控制 Agent 的使用成本？

**参考要点**：
- 模型分级路由：简单任务小模型，复杂任务大模型
- Prompt 精简
- 上下文压缩
- 缓存相同查询
- Token 预算上限
- 提前终止不必要的推理

### Q23: 如何处理生产中的幻觉问题？

**参考要点**：
- RAG 引入可信来源
- 输出格式校验
- 关键信息用工具验证
- 要求标注来源
- 置信度低时拒绝回答或标记不确定
- 高风险输出 Human-in-the-loop 审核

### Q24: Agent 的可观测性应该怎么做？

**参考要点**：
- 全链路日志：请求ID → 推理过程 → 工具调用 → 输出
- 关键指标看板：成功率、延迟（P50/P95/P99）、成本
- 告警规则：错误率 > 5%、P95延迟 > 30s、成本异常
- 定期回放审查，错误分类分析

### Q25: 如何防御 Prompt Injection？

**参考要点**：
- 输入检测：识别注入模式
- 隔离：系统指令与用户输入分层处理
- 输出过滤：防止泄露 System Prompt
- 权限最小化：Agent 只能访问必要工具
- 多层防御：不依赖单一防线
