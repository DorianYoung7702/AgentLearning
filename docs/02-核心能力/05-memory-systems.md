# 记忆系统（Memory Systems）

> 记忆让 Agent 从"无状态问答"变为"有上下文的持续交互"，是构建实用 Agent 的关键能力。

## 1. 为什么 Agent 需要记忆？

- **上下文连贯**：理解多轮对话中的指代和省略
- **知识积累**：记住用户偏好、历史结论
- **避免重复**：不重复执行已完成的操作
- **长期关系**：跨会话维护用户画像

## 2. 记忆类型

### 2.1 三层记忆架构

```
┌──────────────────────────────────────┐
│          感知记忆（Sensory）          │  ← 当前输入的原始信息
├──────────────────────────────────────┤
│         短期记忆（Short-term）        │  ← 当前对话上下文（Context Window）
├──────────────────────────────────────┤
│         长期记忆（Long-term）         │  ← 持久化存储（向量DB/关系DB）
└──────────────────────────────────────┘
```

### 2.2 短期记忆（Working Memory）

- **实现**：LLM 的上下文窗口（Context Window）
- **内容**：当前对话的消息列表
- **容量**：受 Token 限制（4K ~ 128K+）
- **特点**：自动随对话积累，超出窗口则需要裁剪

**管理策略**：
| 策略 | 方法 | 优缺点 |
|------|------|--------|
| 滑动窗口 | 保留最近 N 轮对话 | 简单，但会丢失早期信息 |
| Token 截断 | 保留最近 N 个 Token | 精确控制大小 |
| 摘要压缩 | 将早期对话压缩为摘要 | 保留关键信息，但有信息损失 |
| 重要性筛选 | 只保留重要消息 | 保留关键信息，需判断重要性 |

### 2.3 长期记忆（Long-term Memory）

- **实现**：外部存储（向量数据库、关系数据库、文件系统）
- **内容**：用户偏好、历史交互摘要、学到的知识
- **容量**：理论上无限
- **特点**：需要显式读写操作

**存储方式**：
| 方式 | 适用内容 | 技术 |
|------|---------|------|
| 向量存储 | 语义相关的文本/知识 | Embedding + 向量DB |
| 键值存储 | 结构化的用户偏好 | Redis / JSON |
| 关系数据库 | 结构化的实体关系 | PostgreSQL / MySQL |
| 图数据库 | 复杂知识图谱 | Neo4j |

### 2.4 情景记忆 vs 语义记忆

| 类型 | 说明 | 示例 |
|------|------|------|
| **情景记忆** | 具体的事件和经历 | "用户昨天让我查了天气" |
| **语义记忆** | 抽象的知识和事实 | "用户喜欢简洁的回答风格" |
| **程序记忆** | 如何执行特定任务 | "查天气应该先确认城市" |

## 3. 向量存储与检索

### 3.1 Embedding（嵌入向量）

将文本转为高维向量，语义相近的文本在向量空间中距离更近：

```
"北京天气很好" → [0.12, -0.34, 0.56, ..., 0.78]  (1536维)
"今天的气温" → [0.11, -0.32, 0.55, ..., 0.77]  (语义相近，向量接近)
"代码有bug"  → [-0.45, 0.67, -0.12, ..., 0.34]  (语义不同，向量差异大)
```

**常用 Embedding 模型**：
| 模型 | 厂商 | 维度 | 特点 |
|------|------|------|------|
| text-embedding-3-small | OpenAI | 1536 | 性价比高 |
| text-embedding-3-large | OpenAI | 3072 | 效果最好 |
| bge-large-zh | BAAI | 1024 | 中文优秀，开源 |
| m3e-base | Moka | 768 | 中文，轻量 |

### 3.2 向量数据库

| 数据库 | 特点 | 适用场景 |
|--------|------|---------|
| **Chroma** | 轻量、Python 原生 | 原型开发 |
| **Pinecone** | 全托管、高性能 | 生产环境 |
| **Milvus** | 开源、分布式 | 大规模部署 |
| **Weaviate** | 支持混合搜索 | 需要多种搜索策略 |
| **FAISS** | Facebook 开源、高效 | 本地高性能检索 |
| **pgvector** | PostgreSQL 扩展 | 已有 PG 基础设施 |

### 3.3 相似度搜索

常用距离度量：
- **余弦相似度**：衡量方向相似性，最常用
- **欧氏距离**：衡量绝对距离
- **点积**：归一化向量时等价于余弦相似度

```
余弦相似度 = (A · B) / (||A|| × ||B||)
范围：[-1, 1]，1 表示完全相同
```

## 4. 记忆的读写机制

### 4.1 写入记忆

```
Agent 执行过程中：
  ├── 自动写入：对话历史自动进入短期记忆
  ├── 触发写入：满足条件时写入长期记忆
  │     - 用户明确要求"记住这个"
  │     - 检测到重要信息（偏好、事实）
  │     - 对话结束时总结关键点
  └── 后台写入：异步处理和整理记忆
```

### 4.2 读取记忆

```
收到新消息时：
  1. 自动加载短期记忆（上下文窗口内的历史消息）
  2. 根据当前消息检索长期记忆
     - 语义搜索：Embedding 相似度
     - 关键词搜索：精确匹配
     - 混合搜索：结合两者
  3. 将检索到的记忆注入上下文
  4. LLM 基于完整上下文生成回复
```

### 4.3 记忆更新与遗忘

- **更新**：新信息覆盖旧信息（如用户改了邮箱）
- **合并**：多条相关记忆合并为一条
- **遗忘**：基于时间衰减或重要性淘汰过期记忆
- **冲突处理**：新旧记忆矛盾时以新为准

## 5. 上下文窗口管理（面试重点）

### 5.1 上下文组成

```
System Prompt          [固定开销]
  +
长期记忆检索结果        [动态]
  +
对话历史              [递增]
  +
当前用户输入          [固定]
  +
工具调用结果          [动态]
──────────────────────
= 总上下文 Token 数    [不能超过窗口限制]
```

### 5.2 上下文预算分配策略

```
总窗口大小：32K Token

├── System Prompt:     ~2K   (固定)
├── 长期记忆:          ~4K   (检索 top-k)
├── 工具结果缓冲:      ~8K   (预留给工具输出)
├── 对话历史:          ~14K  (滑动窗口)
└── 输出预留:          ~4K   (给模型生成空间)
```

### 5.3 "Lost in the Middle" 问题

研究发现 LLM 对**上下文开头和结尾**的信息关注度高，**中间部分**容易被忽略。

**应对策略**：
- 重要信息放在上下文的开头或结尾
- 检索结果按相关性重排后，将最相关的放在两端
- 控制上下文长度，避免过长

## 6. 面试高频问题

### Q1: Agent 的短期记忆和长期记忆有什么区别？如何实现？
**要点**：短期记忆 = LLM 上下文窗口，自动维护但容量有限；长期记忆 = 外部存储（向量DB等），容量大但需显式读写。短期处理当前会话，长期跨会话持久化。

### Q2: 如何管理上下文窗口不够用的问题？
**要点**：① 滑动窗口裁剪旧对话 ② 摘要压缩历史 ③ 分配上下文预算 ④ 关键信息存入长期记忆按需检索 ⑤ 工具结果截断。

### Q3: 解释向量检索的工作原理。
**要点**：文本通过 Embedding 模型转为向量 → 存入向量数据库 → 查询时将 query 也转为向量 → 计算余弦相似度找到最相近的文档 → 返回 top-k 结果。

### Q4: 什么是 "Lost in the Middle" 问题？如何解决？
**要点**：LLM 对长上下文中间部分的关注度低于开头和结尾。解决：重要信息放两端、控制上下文长度、使用 Rerank 优化信息排列。

### Q5: 如何设计一个 Agent 的记忆管理策略？
**要点**：① 区分短期/长期记忆 ② 定义写入触发条件 ③ 设计检索策略（语义+关键词混合）④ 上下文预算分配 ⑤ 记忆更新和冲突处理机制 ⑥ 过期淘汰策略。
