# 提示工程（Prompt Engineering）

> 提示工程是 Agent 开发中最核心的技能之一，直接决定 Agent 的行为质量。

## 1. Prompt 的基本结构

### 1.1 Agent System Prompt 标准模板

```
你是 [角色定义]。

## 能力
- [能力1]
- [能力2]

## 可用工具
- [工具1]: [描述]
- [工具2]: [描述]

## 行为规范
- [规则1]
- [规则2]

## 输出格式
[指定输出结构]
```

### 1.2 Prompt 的四个核心要素

| 要素 | 说明 | 示例 |
|------|------|------|
| **角色（Role）** | 定义 Agent 的身份和专业领域 | "你是一位资深数据分析师" |
| **指令（Instruction）** | 明确任务目标和执行方式 | "分析用户上传的数据并生成报告" |
| **上下文（Context）** | 提供背景信息和约束条件 | "用户是非技术人员，避免专业术语" |
| **格式（Format）** | 规定输出结构 | "输出 JSON 格式：{analysis, recommendation}" |

## 2. 核心提示技术

### 2.1 Zero-shot Prompting

直接给出指令，不提供示例：
```
请将以下文本分类为"正面"、"负面"或"中性"：
"这个产品真的太好用了！"
```
- **优点**：简单直接，节省 Token
- **缺点**：复杂任务效果不稳定

### 2.2 Few-shot Prompting

提供若干示例，引导模型学习输出模式：
```
请对用户评论进行情感分类。

示例1：
输入："这个产品真的太好用了！"
输出：正面

示例2：
输入："质量很差，退货了"
输出：负面

请分类：
输入："还行吧，一般般"
输出：
```
- **优点**：通过示例明确任务格式和标准
- **缺点**：消耗较多 Token
- **注意**：示例的质量和多样性很重要

### 2.3 Chain-of-Thought（CoT）思维链

引导模型**逐步推理**，而非直接给出答案：

```
请一步一步思考：

问题：小明有5个苹果，给了小红2个，又买了3个，他现在有几个苹果？

思考过程：
1. 小明初始有 5 个苹果
2. 给了小红 2 个：5 - 2 = 3 个
3. 又买了 3 个：3 + 3 = 6 个

答案：6 个
```

**变体**：
- **Zero-shot CoT**：只需添加 "Let's think step by step"
- **Auto-CoT**：自动生成思维链示例
- **在 Agent 中的应用**：让 Agent 在每步行动前先"思考"原因

### 2.4 Self-Consistency

多次生成，取多数一致的结果：
```
对同一问题进行 N 次 CoT 推理
→ 得到 N 个答案
→ 投票选出出现次数最多的答案
```
- 提高推理可靠性，但增加成本

### 2.5 ReAct Prompt

将推理和行动交织在 Prompt 中（Agent 核心）：

```
Answer the following questions using the available tools.

Tools:
- search(query): Search the web
- calculator(expression): Calculate math expressions

Question: {question}

Thought: I need to think about what to do
Action: tool_name(arguments)
Observation: [tool result]
... (repeat)
Thought: I now know the answer
Final Answer: [answer]
```

## 3. 高级 Prompt 技术

### 3.1 Tree of Thoughts (ToT)

将推理过程组织为**树形结构**，探索多条路径：

```
                  问题
              /    |    \
          思路A  思路B  思路C
          / \     |     / \
        A1  A2   B1   C1  C2
        ↓        ↓         ↓
      评估     评估       评估
        ↓        
    最佳方案
```

- 每步生成多个候选想法
- 对每个候选进行评估打分
- 选择最优路径继续
- **适用场景**：需要创造性解决方案或复杂推理的任务

### 3.2 Reflexion（反思）

让 Agent 在失败后自我反思并改进：

```
第1次尝试：
  行动 → 结果（失败）
  反思："我失败的原因是...下次应该..."
  
第2次尝试（利用反思经验）：
  行动 → 结果（成功）
```

### 3.3 Structured Output Prompt

强制模型输出结构化数据（Agent 工具调用的基础）：

```
请以以下 JSON 格式回复，不要包含其他文字：

{
  "action": "工具名称",
  "parameters": {
    "param1": "值1",
    "param2": "值2"
  },
  "reasoning": "选择该工具的原因"
}
```

## 4. Agent Prompt 设计最佳实践

### 4.1 System Prompt 设计原则

1. **角色清晰**：明确 Agent 的身份、能力边界
2. **工具描述准确**：每个工具的功能、参数、返回值都要描述清楚
3. **行为约束明确**：什么该做、什么不该做
4. **输出格式固定**：方便程序解析（JSON/XML）
5. **错误处理指引**：工具调用失败时的行为

### 4.2 常见 Prompt 陷阱

| 陷阱 | 问题 | 解决方案 |
|------|------|---------|
| 指令模糊 | 模型理解偏差 | 使用具体、无歧义的表述 |
| 角色冲突 | 多个矛盾的要求 | 明确优先级 |
| 格式不一致 | 输出解析失败 | 提供严格的格式模板和示例 |
| 上下文过长 | 关键信息被忽略 | 将重要指令放在 Prompt 开头和结尾 |
| 缺少边界条件 | Agent 在边缘情况出错 | 枚举常见异常情况的处理方式 |

### 4.3 Prompt 迭代优化流程

```
1. 编写初版 Prompt
2. 用测试用例跑测
3. 分析失败案例
4. 针对性修改 Prompt
5. 回归测试（确保修改不破坏已有能力）
6. 重复 2-5
```

## 5. 面试高频问题

### Q1: 什么是 Chain-of-Thought？为什么它能提升 LLM 推理能力？
**要点**：CoT 通过引导模型逐步分解问题，将复杂推理拆成简单的中间步骤。原理类似于人类的"草稿纸"思考，让模型在生成中间推理过程时，为后续推理提供了更好的上下文。

### Q2: 在 Agent 中，System Prompt 应该包含哪些关键信息？
**要点**：① 角色定义 ② 能力和限制 ③ 可用工具列表及详细描述 ④ 输出格式要求 ⑤ 行为约束和安全规则 ⑥ 错误处理指引。

### Q3: Few-shot 和 Zero-shot 如何选择？
**要点**：任务简单明确用 Zero-shot 节省 Token；输出格式严格或任务复杂用 Few-shot 提供示范。Agent 场景中工具调用通常用 Few-shot 来确保格式正确。

### Q4: 如何评估和优化 Prompt 的效果？
**要点**：① 建立测试集覆盖典型和边缘情况 ② 定义评估指标（准确率、格式正确率）③ A/B 测试不同版本 ④ 分析失败案例针对性优化 ⑤ 回归测试防止退化。

### Q5: 对比 CoT、ToT、Reflexion 三种推理策略。
**要点**：
- CoT：线性逐步推理，简单高效
- ToT：树形分支探索，适合需要发散思考的问题，成本更高
- Reflexion：加入失败后的反思循环，适合试错型任务
