# 模型微调（Fine-tuning）

> 微调是让通用 LLM 适配特定任务的核心技术，大厂面试必考。

## 1. 为什么需要微调？

### Prompt Engineering vs Fine-tuning vs RAG

| 方法 | 适用场景 | 成本 | 效果 |
|------|---------|------|------|
| **Prompt Engineering** | 通用任务，格式简单 | 低 | 受限于模型能力 |
| **RAG** | 需要外部知识，知识频繁更新 | 中 | 知识注入好，风格不变 |
| **Fine-tuning** | 需要特定风格/格式/推理模式 | 高 | 能改变模型行为 |
| **三者结合** | 复杂生产场景 | 最高 | 最佳 |

### 微调的典型目标
- 学习特定输出格式（如 JSON、工具调用格式）
- 学习领域知识（医疗、法律、金融术语）
- 学习交互风格（简洁、专业、活泼）
- 提升特定任务性能（分类、摘要、翻译）
- 减少幻觉（在特定领域）
- 减小模型体积（用小模型微调替代大模型）

## 2. 微调方法分类

### 2.1 全参数微调（Full Fine-tuning）

更新模型**所有参数**：

```
原始模型参数 (7B) → 训练 → 更新后的模型参数 (7B)
```

- **优点**：效果上限最高
- **缺点**：需要大量显存（7B 模型需要 ~60GB）、训练慢、存储成本高
- **适用**：资源充足、追求极致效果

### 2.2 参数高效微调（PEFT - Parameter Efficient Fine-Tuning）

只更新**少量参数**，冻结大部分原始参数：

```
原始模型参数 (7B, 冻结) + 新增参数 (~0.1B, 可训练)
→ 训练只更新新增参数
→ 推理时合并
```

主要方法：

| 方法 | 原理 | 参数量 | 效果 |
|------|------|--------|------|
| **LoRA** | 低秩矩阵分解 | 原始的 0.1%-1% | 接近全参数 |
| **QLoRA** | LoRA + 量化 | 同上 | 接近 LoRA |
| **Prefix Tuning** | 学习前缀向量 | 极少 | 中等 |
| **P-Tuning v2** | 每层学习前缀 | 少 | 较好 |
| **Adapter** | 插入适配器层 | 少 | 较好 |

## 3. LoRA 详解（面试重点）

### 3.1 核心原理

**思想**：模型微调时的参数变化矩阵 ΔW 是**低秩**的，可以分解为两个小矩阵的乘积。

```
原始权重矩阵: W₀ ∈ R^(d×d)    例如 4096×4096

传统微调: W = W₀ + ΔW           ΔW 是 4096×4096 = 16M 参数

LoRA: W = W₀ + B·A
  A ∈ R^(r×d)    例如 8×4096 = 32K 参数
  B ∈ R^(d×r)    例如 4096×8 = 32K 参数
  总计: 64K 参数   (vs 16M，减少 250 倍)
```

**可视化**：
```
        ┌─────────┐
  x ──→ │  W₀     │ ──→ h (冻结，不更新)
  │     │(d × d)  │
  │     └─────────┘
  │
  │     ┌───┐   ┌───┐
  └───→ │ A │ → │ B │ ──→ Δh (可训练)
        │r×d│   │d×r│
        └───┘   └───┘
        
最终输出: h + α·Δh   (α 是缩放因子)
```

### 3.2 关键超参数

| 参数 | 说明 | 典型值 | 影响 |
|------|------|--------|------|
| **rank (r)** | 低秩矩阵的秩 | 8, 16, 32, 64 | 越大参数越多，拟合能力越强 |
| **alpha (α)** | 缩放因子 | 16, 32 | 控制 LoRA 的影响强度 |
| **target_modules** | 应用 LoRA 的层 | q_proj, v_proj | 哪些权重矩阵加 LoRA |
| **dropout** | LoRA 层的 Dropout | 0.05-0.1 | 防止过拟合 |

### 3.3 LoRA 应用到哪些层？

```
Transformer Layer:
  ├── Self-Attention
  │   ├── q_proj  ← 通常加 LoRA ✅
  │   ├── k_proj  ← 可选
  │   ├── v_proj  ← 通常加 LoRA ✅
  │   └── o_proj  ← 可选
  ├── MLP
  │   ├── gate_proj ← 可选
  │   ├── up_proj   ← 可选
  │   └── down_proj ← 可选
  └── LayerNorm ← 通常不加
```

- 最基础：只加 q_proj 和 v_proj
- 更好的效果：q/k/v/o 全加
- 最佳效果：全部线性层都加（接近全参数微调效果）

### 3.4 LoRA 的优势

1. **显存大幅减少**：7B 模型全参微调需 ~60GB，LoRA 仅需 ~16GB
2. **训练速度快**：更新参数少
3. **存储高效**：LoRA 权重只有几十 MB
4. **多任务切换**：同一基座模型 + 不同 LoRA 适配器
5. **可合并**：推理时将 LoRA 合并回原始权重，无额外推理开销

## 4. QLoRA 详解

### 4.1 核心思想

**QLoRA = 量化基座模型 + LoRA 微调**

```
普通 LoRA:
  基座模型 (FP16, ~14GB for 7B) + LoRA 参数 → 需要 ~16GB 显存

QLoRA:
  基座模型 (4-bit 量化, ~4GB for 7B) + LoRA 参数 (FP16) → 需要 ~6GB 显存
```

### 4.2 关键技术

1. **4-bit NormalFloat (NF4)**：专为正态分布权重设计的量化格式
2. **Double Quantization**：对量化常数再次量化，进一步节省显存
3. **Paged Optimizers**：用 CPU 内存分页管理优化器状态

### 4.3 QLoRA vs LoRA

| 维度 | LoRA | QLoRA |
|------|------|-------|
| 基座精度 | FP16 | 4-bit |
| 显存需求 | ~16GB (7B) | ~6GB (7B) |
| 训练速度 | 快 | 稍慢（量化/反量化开销） |
| 效果 | 好 | 接近 LoRA |
| 适用 | 有 A100 等高端卡 | 消费级显卡（RTX 3090/4090） |

## 5. 微调数据准备

### 5.1 数据格式

**指令微调格式（Alpaca 风格）**：
```json
{
  "instruction": "将以下文本翻译成英文",
  "input": "今天天气很好",
  "output": "The weather is nice today"
}
```

**对话格式（ShareGPT 风格）**：
```json
{
  "conversations": [
    {"role": "system", "content": "你是一个翻译助手"},
    {"role": "user", "content": "翻译：今天天气很好"},
    {"role": "assistant", "content": "The weather is nice today"}
  ]
}
```

**工具调用格式**：
```json
{
  "conversations": [
    {"role": "user", "content": "北京今天天气怎么样？"},
    {"role": "assistant", "content": null, "tool_calls": [
      {"name": "get_weather", "arguments": {"city": "北京"}}
    ]},
    {"role": "tool", "content": "{\"temp\": 25, \"condition\": \"晴\"}"},
    {"role": "assistant", "content": "北京今天天气晴朗，气温25°C。"}
  ]
}
```

### 5.2 数据质量要求

| 维度 | 要求 |
|------|------|
| 准确性 | 答案必须正确 |
| 多样性 | 覆盖各种场景和问法 |
| 一致性 | 相似问题回答风格一致 |
| 数量 | SFT 通常 1K-100K 条 |
| 格式 | 严格遵循模板 |

### 5.3 数据增强技术

- **Self-Instruct**：用 LLM 生成训练数据
- **Evol-Instruct**：渐进式增加指令复杂度
- **回译增强**：翻译后再翻译回来生成变体
- **拒绝采样**：生成多个回答，选择最优的

## 6. 微调训练实践

### 6.1 常用工具

| 工具 | 说明 |
|------|------|
| **Hugging Face Transformers + PEFT** | 最通用 |
| **LLaMA-Factory** | 中文社区最流行，GUI 界面 |
| **Axolotl** | 配置驱动，支持多种方法 |
| **Unsloth** | 速度最快，显存最省 |
| **DeepSpeed** | 分布式训练 |

### 6.2 训练超参数

| 参数 | 典型值 | 说明 |
|------|--------|------|
| learning_rate | 1e-4 ~ 2e-5 | LoRA 通常比全参高 |
| epochs | 2-5 | 避免过拟合 |
| batch_size | 4-32 | 受限于显存 |
| warmup_ratio | 0.03-0.1 | 学习率预热 |
| max_length | 512-2048 | 最大序列长度 |
| gradient_accumulation | 4-16 | 模拟大 batch |

### 6.3 过拟合检测和预防

```
检测信号：
- 训练 loss 持续降低但验证 loss 上升
- 模型输出变得固化（总是重复训练数据中的内容）
- 对训练集以外的输入效果变差

预防措施：
- 减少 epoch 数
- 增加数据量和多样性
- 使用 dropout
- 降低 LoRA rank
- 早停（Early Stopping）
```

## 7. 微调评估

### 7.1 评估方法

| 方法 | 适用场景 |
|------|---------|
| **困惑度（Perplexity）** | 通用指标，越低越好 |
| **任务指标** | 准确率/F1/BLEU/ROUGE 等 |
| **人工评估** | 开放式任务 |
| **LLM-as-Judge** | 可扩展的质量评估 |
| **A/B 测试** | 生产环境对比 |

### 7.2 对比基线

```
评估链路：
  基座模型（零样本）
    vs 基座 + Prompt 优化
    vs 基座 + RAG
    vs 微调模型
    vs 微调 + RAG
```

## 8. 面试高频问题

### Q1: 解释 LoRA 的原理。为什么有效？
**要点**：微调时权重变化 ΔW 是低秩的，可分解为两个小矩阵 B·A 的乘积。r=8 时参数量只有全参的 0.1%，但效果接近全参。有效原因：预训练模型已学到丰富表示，微调只需在低维子空间中调整。

### Q2: LoRA 的 rank 应该怎么选？
**要点**：r=8 适合简单任务；r=16-32 适合中等复杂度；r=64+ 接近全参效果但成本增高。选择策略：从小 rank 开始，效果不够再增大。Alpha 通常设为 rank 的 2 倍。

### Q3: QLoRA 相比 LoRA 做了什么优化？
**要点**：将基座模型量化为 4-bit（NF4 格式），LoRA 参数保持 FP16。加上 Double Quantization 和 Paged Optimizers。7B 模型从需要 16GB 降到 6GB，可在消费级显卡上训练。

### Q4: 什么时候该用微调？什么时候用 RAG？
**要点**：微调适合学习风格、格式、推理模式；RAG 适合注入知识、需要引用来源、知识频繁更新。两者可结合：微调让模型更好地利用检索结果。

### Q5: 如何准备微调数据？
**要点**：① 定义任务和输出格式 ② 收集或构造种子数据 ③ 用 LLM 辅助扩增（Self-Instruct）④ 人工审核质量 ⑤ 确保多样性和准确性 ⑥ 通常需要 1K-100K 条高质量数据。

### Q6: 微调后模型效果变差（灾难性遗忘）怎么办？
**要点**：① 使用 LoRA 而非全参微调（影响范围小）② 混入通用数据一起训练 ③ 减少训练轮次 ④ 降低学习率 ⑤ 评估通用能力指标的退化程度。

### Q7: 对比 LoRA、Prefix Tuning、Adapter 三种 PEFT 方法。
**要点**：
- LoRA：修改注意力权重矩阵，效果最好，最流行
- Prefix Tuning：在输入前加可学习的虚拟 Token，不改模型结构
- Adapter：在 Transformer 层间插入小网络
- 实践中 LoRA 是首选，效果最好且实现成熟
