# 工具使用与 Function Calling

> 工具使用能力是 Agent 区别于普通 ChatBot 的核心特征，让 LLM 从"只能说"变为"能做事"。

## 1. 为什么 Agent 需要工具？

LLM 的天然局限性需要工具来弥补：

| LLM 局限 | 工具解决方案 |
|----------|-------------|
| 知识有截止日期 | 搜索引擎获取最新信息 |
| 数学计算不可靠 | 计算器/代码执行器 |
| 无法访问私有数据 | 数据库查询/API 调用 |
| 无法影响外部世界 | 邮件发送/文件操作/系统命令 |
| 无法处理多模态 | 图像识别/语音转文字 |

## 2. Function Calling 机制

### 2.1 工作流程

```
1. 定义工具（函数签名 + 描述）
   ↓
2. 用户发送请求
   ↓
3. LLM 分析请求，决定是否调用工具
   ↓
4a. 不需要工具 → 直接回复
4b. 需要工具 → 生成工具调用（函数名 + 参数）
   ↓
5. 应用层执行工具调用，获取结果
   ↓
6. 将结果返回给 LLM
   ↓
7. LLM 基于结果生成最终回复
```

### 2.2 工具定义格式（OpenAI 风格）

```json
{
  "type": "function",
  "function": {
    "name": "get_weather",
    "description": "获取指定城市的当前天气信息",
    "parameters": {
      "type": "object",
      "properties": {
        "city": {
          "type": "string",
          "description": "城市名称，如'北京'"
        },
        "unit": {
          "type": "string",
          "enum": ["celsius", "fahrenheit"],
          "description": "温度单位"
        }
      },
      "required": ["city"]
    }
  }
}
```

### 2.3 LLM 返回的工具调用

```json
{
  "role": "assistant",
  "content": null,
  "tool_calls": [
    {
      "id": "call_abc123",
      "type": "function",
      "function": {
        "name": "get_weather",
        "arguments": "{\"city\": \"北京\", \"unit\": \"celsius\"}"
      }
    }
  ]
}
```

### 2.4 工具结果返回

```json
{
  "role": "tool",
  "tool_call_id": "call_abc123",
  "content": "{\"temperature\": 25, \"condition\": \"晴\", \"humidity\": 40}"
}
```

## 3. 工具设计原则

### 3.1 好的工具描述

工具描述的质量直接影响 LLM 的调用决策：

**好的描述** ✅：
```
name: search_database
description: 在知识库中搜索相关文档。输入搜索查询字符串，返回最相关的
前5条文档摘要。适用于需要查找公司内部文档、产品手册、FAQ的场景。
不适用于搜索实时互联网信息。
```

**差的描述** ❌：
```
name: search
description: 搜索功能
```

### 3.2 工具设计最佳实践

1. **职责单一**：每个工具只做一件事
2. **命名清晰**：`search_company_docs` 优于 `search`
3. **参数明确**：类型、范围、默认值、必填/选填
4. **描述详尽**：写明用途、适用场景、不适用场景、返回格式
5. **错误处理**：返回有意义的错误信息，而非让程序崩溃
6. **幂等性**：同样的输入产生同样的结果（尤其是查询类工具）

### 3.3 工具粒度

| 粒度 | 示例 | 优点 | 缺点 |
|------|------|------|------|
| 太粗 | `do_everything(task)` | 调用简单 | LLM 难以正确使用 |
| 适中 | `search_docs(query)` | 语义清晰 | — |
| 太细 | `open_db()`, `run_query()`, `close_db()` | 灵活 | 步骤多，LLM 容易出错 |

## 4. 并行工具调用

### 4.1 概念

LLM 可以在一次响应中返回多个工具调用：

```json
{
  "tool_calls": [
    {"function": {"name": "get_weather", "arguments": "{\"city\": \"北京\"}"}},
    {"function": {"name": "get_weather", "arguments": "{\"city\": \"上海\"}"}}
  ]
}
```

### 4.2 处理策略
- **并行执行**：无依赖关系的工具可同时执行，减少延迟
- **串行执行**：有依赖关系时必须按序执行
- **部分失败处理**：某个工具失败不应阻塞其他工具

## 5. 常见工具类型

### 5.1 信息检索类
- **搜索引擎**：Google/Bing API
- **知识库搜索**：向量数据库查询
- **数据库查询**：SQL 执行
- **文档读取**：PDF/Word 解析

### 5.2 计算与执行类
- **代码执行器**：Python/JavaScript 沙箱
- **计算器**：数学计算
- **数据分析**：pandas 操作

### 5.3 外部交互类
- **API 调用**：第三方服务集成
- **邮件发送**：通知和沟通
- **文件操作**：创建/修改/删除文件

### 5.4 内部管理类
- **记忆存储**：保存重要信息到长期记忆
- **任务管理**：创建/更新子任务
- **自我反思**：调用反思工具评估当前状态

## 6. 工具调用的安全性

### 6.1 风险分类

| 风险等级 | 工具类型 | 示例 |
|---------|---------|------|
| 低 | 只读查询 | 搜索、天气查询 |
| 中 | 数据写入 | 更新数据库、发送消息 |
| 高 | 系统操作 | 执行代码、删除文件 |

### 6.2 安全措施

1. **权限控制**：不同工具设置不同权限级别
2. **人机协同（Human-in-the-loop）**：高风险操作需要用户确认
3. **沙箱执行**：代码执行在隔离环境中运行
4. **输入验证**：校验 LLM 生成的参数是否合法
5. **速率限制**：防止工具被恶意滥用
6. **审计日志**：记录所有工具调用便于追溯

## 7. 工具调用失败处理

```
工具调用失败
    ├── 参数错误 → 让 LLM 修正参数重试
    ├── 工具不可用 → 使用替代工具或直接回答
    ├── 超时 → 重试或告知用户
    └── 结果为空 → 换一种查询方式或调整策略
```

### 最大重试次数

通常设置 2-3 次重试上限，防止无限循环。

## 8. 面试高频问题

### Q1: 解释 Function Calling 的完整流程。
**要点**：定义工具 Schema → 用户提问 → LLM 判断是否需要工具 → 生成调用（函数名+参数 JSON）→ 应用层执行 → 结果返回 LLM → LLM 生成最终回答。

### Q2: 如何设计一个好的工具描述？
**要点**：① 明确工具的功能和用途 ② 说明适用和不适用场景 ③ 参数类型和约束要清晰 ④ 描述返回值格式 ⑤ 给出使用示例。

### Q3: Agent 工具调用失败了怎么办？
**要点**：① 区分错误类型（参数错误/工具不可用/超时）② 参数错误让 LLM 修正重试 ③ 设置最大重试次数 ④ 提供降级方案（替代工具或直接回答）⑤ 向用户透明地报告无法完成的操作。

### Q4: 如何保证工具调用的安全性？
**要点**：① 权限分级 ② Human-in-the-loop 确认高危操作 ③ 沙箱执行代码 ④ 参数验证 ⑤ 速率限制 ⑥ 审计日志。

### Q5: 并行工具调用和串行工具调用各自适用什么场景？
**要点**：并行 — 多个独立查询（如同时查多个城市天气）；串行 — 有依赖关系（如先搜索再基于结果计算）。并行可降低延迟但需处理部分失败。
