# 检索增强生成（RAG）

> RAG 是让 Agent 拥有"外部知识"的核心技术，解决 LLM 知识截止和幻觉问题。

## 1. 什么是 RAG？

**RAG（Retrieval-Augmented Generation）** = 先检索相关知识，再让 LLM 基于检索结果生成回答。

```
用户问题 → [检索器] → 找到相关文档 → [LLM] → 基于文档生成回答
                ↑
           知识库（向量DB）
```

### 为什么需要 RAG？

| 问题 | RAG 如何解决 |
|------|-------------|
| 模型知识有截止日期 | 检索最新数据 |
| 缺少私有/领域知识 | 接入企业知识库 |
| 幻觉问题 | 基于真实文档回答，可引用来源 |
| 微调成本高 | 无需重新训练模型 |

## 2. RAG 完整流程

### 2.1 离线索引阶段

```
原始文档
  │
  ▼
[文档加载] ← PDF/Word/HTML/Markdown 解析
  │
  ▼
[文本切分] ← Chunking 策略
  │
  ▼
[向量化]   ← Embedding 模型
  │
  ▼
[存储]     → 向量数据库
```

### 2.2 在线查询阶段

```
用户问题
  │
  ▼
[查询处理] ← 查询改写/扩展
  │
  ▼
[检索]     ← 向量搜索 + 关键词搜索
  │
  ▼
[重排序]   ← Reranker 精排
  │
  ▼
[上下文组装] ← 拼接 Prompt
  │
  ▼
[LLM 生成] → 回答（附来源引用）
```

## 3. 文本切分策略（Chunking）

### 3.1 为什么需要切分？

- 文档通常很长，超过 Embedding 模型的处理能力
- 细粒度的 Chunk 检索更精准
- 控制注入 LLM 的上下文长度

### 3.2 常见切分方法

| 方法 | 说明 | 适用场景 |
|------|------|---------|
| **固定大小** | 按字符/Token 数切分 | 通用，简单快速 |
| **递归切分** | 按段落→句子→词逐层切分 | 保持语义完整性 |
| **语义切分** | 基于语义相似度判断断点 | 精准但计算量大 |
| **文档结构切分** | 按标题/章节切分 | Markdown/HTML 等结构化文档 |
| **滑动窗口** | 重叠切分，相邻 Chunk 有重叠 | 防止关键信息被切断 |

### 3.3 Chunk 大小选择

| Chunk 大小 | 优点 | 缺点 |
|------------|------|------|
| 小（100-200 Token） | 检索精准 | 缺少上下文 |
| 中（300-500 Token） | 平衡精准和上下文 | 通用选择 |
| 大（500-1000 Token） | 上下文丰富 | 可能包含无关信息 |

**最佳实践**：通常 256-512 Token，overlap 50-100 Token。

### 3.4 Chunk 增强技术

- **Parent-Child Chunk**：检索用小 Chunk，返回时用大 Chunk（包含上下文）
- **Chunk 元数据**：给每个 Chunk 附加标题、来源、日期等元数据
- **Chunk 摘要**：为每个 Chunk 生成摘要，用摘要做检索

## 4. 检索策略

### 4.1 向量检索（Semantic Search）

```
Query → Embedding → 在向量空间找 Top-K 最近邻
```
- 语义匹配，能理解同义词和近义表达
- 可能遗漏关键词精确匹配的场景

### 4.2 关键词检索（Keyword Search）

```
Query → 分词 → BM25/TF-IDF 匹配
```
- 精确匹配特定术语、代号
- 不理解语义

### 4.3 混合检索（Hybrid Search）

```
Query → [向量检索] → 结果A (语义相关)
     → [关键词检索] → 结果B (精确匹配)
     → 融合排序 → 最终结果
```

融合方法：
- **RRF（Reciprocal Rank Fusion）**：基于排名的简单融合
- **加权融合**：给两种结果分配权重

### 4.4 多路召回

```
Query → [向量检索]     → 候选集1
     → [关键词检索]    → 候选集2  → 合并 → Rerank → Top-K
     → [知识图谱检索]  → 候选集3
```

## 5. Rerank（重排序）

### 5.1 为什么需要 Rerank？

- 初始检索是粗排（效率优先），可能不够精准
- Rerank 用更强的模型对候选集进行精细排序
- 尤其在混合检索后效果提升明显

### 5.2 常用 Reranker

| 模型 | 说明 |
|------|------|
| Cohere Rerank | 商用，效果好 |
| bge-reranker | 开源，中文友好 |
| Cross-Encoder | 传统方法，精准但慢 |

### 5.3 Rerank 流程

```
检索 Top-50 候选文档
  ↓
Reranker 对每个文档与 Query 进行精细评分
  ↓
按新分数重排
  ↓
取 Top-5 返回
```

## 6. 查询优化技术

### 6.1 查询改写（Query Rewriting）

用 LLM 将用户的原始查询改写为更适合检索的形式：
```
原始：公司有什么福利？
改写：员工福利政策 薪酬待遇 补贴 假期
```

### 6.2 查询扩展（Query Expansion）

生成多个相关查询，扩大检索范围：
```
原始：RAG 的优缺点
扩展1：RAG 检索增强生成的优点是什么
扩展2：RAG 的局限性和挑战
扩展3：RAG vs Fine-tuning 对比
```

### 6.3 HyDE（假设文档嵌入）

让 LLM 先"假设"一个答案文档，用这个假设文档做检索：
```
Query: "如何优化 RAG 的检索效果？"
  ↓
LLM 生成假设答案: "优化 RAG 检索效果可以从以下几个方面..."
  ↓
用假设答案的 Embedding 做检索（比原始 Query 更接近真实文档）
```

### 6.4 Step-back Prompting

先问一个更高层次的问题，获取背景知识，再回答原始问题。

## 7. RAG 评估指标

### 7.1 检索质量

| 指标 | 说明 |
|------|------|
| **Recall@K** | Top-K 结果中包含正确文档的比例 |
| **Precision@K** | Top-K 结果中相关文档的占比 |
| **MRR** | 第一个相关文档的排名倒数 |
| **NDCG** | 考虑排名位置的评估指标 |

### 7.2 生成质量

| 指标 | 说明 |
|------|------|
| **Faithfulness** | 回答是否忠实于检索到的文档 |
| **Relevance** | 回答是否切题 |
| **Completeness** | 回答是否完整覆盖要点 |

### 7.3 评估框架

- **RAGAS**：自动化 RAG 评估框架
- **TruLens**：基于反馈函数的评估

## 8. Advanced RAG 技术

### 8.1 Self-RAG

Agent 自主决定是否需要检索：
```
收到问题 → 判断是否需要检索
  ├── 不需要 → 直接回答
  └── 需要 → 检索 → 评估检索结果是否有用
              ├── 有用 → 基于结果回答
              └── 无用 → 换策略重试或直接回答
```

### 8.2 Corrective RAG（CRAG）

检索后评估文档质量，必要时触发额外检索：
```
检索 → 评估文档质量
  ├── 高质量 → 直接使用
  ├── 模糊   → 额外检索补充
  └── 低质量 → 丢弃，用 Web 搜索替代
```

### 8.3 Graph RAG

结合知识图谱的 RAG，先用图谱找到相关实体和关系，再检索文档。

## 9. 面试高频问题

### Q1: 解释 RAG 的完整流程。
**要点**：离线阶段：文档加载 → 切分 → 向量化 → 存储。在线阶段：查询处理 → 检索 → Rerank → 上下文组装 → LLM 生成回答。

### Q2: RAG vs Fine-tuning，各自适用什么场景？
**要点**：RAG 适合知识频繁更新、需要引用来源、数据量不足以微调的场景；Fine-tuning 适合学习特定风格、格式、推理模式。两者可组合使用。

### Q3: 如何选择 Chunk 大小？
**要点**：通常 256-512 Token，overlap 50-100。小 Chunk 检索精准但缺上下文，大 Chunk 上下文丰富但有噪声。需根据数据特点实验调优。

### Q4: 什么是混合检索？为什么比单一检索好？
**要点**：混合检索结合向量检索（语义匹配）和关键词检索（精确匹配），用 RRF 等方法融合。向量弥补关键词的语义不足，关键词弥补向量的精确性不足。

### Q5: 如何评估 RAG 系统的效果？
**要点**：分两层评估 — 检索质量（Recall@K、MRR）和生成质量（Faithfulness、Relevance）。使用 RAGAS 等框架自动化评估。重点关注 Faithfulness（不产生幻觉）。
