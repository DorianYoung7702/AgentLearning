# 多智能体系统（Multi-Agent Systems）

> 当单个 Agent 难以独立完成复杂任务时，多 Agent 协作是自然的进化方向。

## 1. 为什么需要多 Agent？

### 单 Agent 的局限
- **能力瓶颈**：一个 Prompt 难以覆盖所有专业知识
- **上下文限制**：复杂任务超出单次上下文窗口
- **可靠性**：单点失败无法纠错
- **并行度**：单 Agent 只能串行工作

### 多 Agent 的优势
- **专业分工**：每个 Agent 专注一个领域
- **并行执行**：独立子任务可同时处理
- **相互校验**：Agent 之间可审查对方输出
- **模块化**：便于维护和扩展

## 2. 多 Agent 架构模式

### 2.1 中心化架构（Orchestrator 模式）

```
              ┌──────────┐
              │ Orchestrator │ ← 负责分配和协调
              │  (编排器)    │
              └──┬───┬───┬──┘
                 │   │   │
         ┌───┐ ┌───┐ ┌───┐
         │A1 │ │A2 │ │A3 │  ← 执行 Agent
         └───┘ └───┘ └───┘
```

- **Orchestrator**：接收任务、分解、分配给子 Agent、汇总结果
- **子 Agent**：各自执行分配的子任务
- **优点**：控制力强，流程清晰
- **缺点**：Orchestrator 是瓶颈和单点故障

### 2.2 去中心化架构（Peer-to-Peer）

```
         ┌───┐ ←──→ ┌───┐
         │A1 │      │A2 │
         └───┘ ←──→ └───┘
            ↕          ↕
         ┌───┐ ←──→ ┌───┐
         │A3 │      │A4 │
         └───┘      └───┘
```

- Agent 之间直接通信协作
- **优点**：无单点故障，灵活
- **缺点**：协调复杂，可能出现冲突

### 2.3 层级架构（Hierarchical）

```
              [Manager]
              /       \
        [Team Lead A] [Team Lead B]
         /     \         /     \
      [W1]   [W2]    [W3]   [W4]
```

- 多层管理结构，每层负责不同粒度的任务
- 适合大型复杂项目

### 2.4 流水线架构（Pipeline）

```
[Agent1] → [Agent2] → [Agent3] → [Agent4]
 研究         分析       写作       审校
```

- Agent 按顺序处理，每个 Agent 处理一个环节
- 适合有明确阶段的任务

## 3. 多 Agent 通信模式

### 3.1 消息传递

```python
# 基本消息格式
{
    "from": "researcher_agent",
    "to": "writer_agent",
    "content": "以下是搜集到的资料...",
    "type": "task_result",
    "metadata": {"task_id": "001"}
}
```

### 3.2 共享状态

```
所有 Agent 共享一个全局状态对象：

State = {
    "task": "撰写市场分析报告",
    "research_data": [...],      ← Researcher 写入
    "analysis_result": {...},    ← Analyst 写入
    "draft": "...",              ← Writer 写入
    "review_feedback": "...",    ← Reviewer 写入
    "status": "reviewing"
}
```

### 3.3 对话/辩论

Agent 之间通过对话交换观点：
```
Agent A（正方）：我认为应该使用方案1，因为...
Agent B（反方）：方案1有一个问题...方案2更好
Agent A：你说的对，但方案2的成本更高...
Judge Agent：综合两方观点，建议采用方案1并加上方案2的安全措施
```

## 4. 角色设计

### 4.1 常见角色类型

| 角色 | 职责 | 示例 System Prompt 关键词 |
|------|------|-------------------------|
| **Planner** | 任务分解和规划 | "你负责将任务分解为子任务" |
| **Researcher** | 信息搜集 | "你负责搜索和整理相关信息" |
| **Coder** | 代码编写 | "你是一位资深程序员" |
| **Reviewer** | 质量审查 | "你负责审查并提出改进意见" |
| **Writer** | 内容创作 | "你负责撰写高质量文本" |
| **Critic** | 批判性评估 | "你负责找出方案中的问题" |

### 4.2 角色设计原则

1. **职责清晰**：每个 Agent 有明确的专业定位
2. **能力互补**：Agent 之间能力不重叠但互补
3. **工具匹配**：每个 Agent 只拥有其职责需要的工具
4. **Prompt 差异化**：不同角色使用不同的 System Prompt

## 5. 任务分配与协调

### 5.1 任务分配策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| 静态分配 | 预定义任务-Agent 映射 | 流程固定的任务 |
| 动态分配 | Orchestrator 根据任务内容匹配 Agent | 灵活多变的任务 |
| 竞标机制 | Agent 自报能力，竞争任务 | Agent 能力有重叠时 |
| 轮询 | 平均分配 | 同质 Agent 并行处理 |

### 5.2 协调机制

- **同步协调**：等所有 Agent 完成当前步骤后统一推进
- **异步协调**：Agent 完成后立即触发下游
- **条件触发**：满足特定条件时触发特定 Agent
- **投票决策**：多 Agent 投票决定下一步行动

## 6. 多 Agent 的挑战

### 6.1 核心挑战

| 挑战 | 说明 | 解决思路 |
|------|------|---------|
| 通信开销 | Agent 间消息传递消耗 Token | 压缩消息、减少冗余通信 |
| 冲突处理 | Agent 意见不一致 | 仲裁机制、优先级规则 |
| 循环问题 | Agent 互相推诿或无限循环 | 设置最大轮次、超时机制 |
| 上下文同步 | 保持各 Agent 的信息一致 | 共享状态管理 |
| 调试困难 | 多 Agent 交互难以追踪 | 完善的日志和可视化 |
| 成本控制 | 多 Agent = 多次 LLM 调用 | 优化通信频率、用小模型处理简单任务 |

### 6.2 死锁预防

```
防止死锁的策略：
1. 设置全局最大轮次（如最多10轮对话）
2. 超时自动终止
3. 检测重复消息并中断循环
4. Orchestrator 有权强制终止任何 Agent
```

## 7. 典型多 Agent 应用场景

### 7.1 软件开发团队

```
Product Manager → 需求分析
      ↓
Architect → 系统设计
      ↓
Coder → 代码编写 ←→ Reviewer → 代码审查
      ↓
Tester → 测试用例编写和执行
```

### 7.2 研究分析

```
Researcher (多个) → 并行搜集不同方面的信息
      ↓
Analyst → 整合分析
      ↓
Writer → 撰写报告
      ↓
Critic → 审查质量
```

### 7.3 客服系统

```
Router Agent → 意图分类
  ├── FAQ Agent → 常见问题
  ├── Order Agent → 订单查询
  ├── Tech Agent → 技术支持
  └── Escalation Agent → 人工转接
```

## 8. 面试高频问题

### Q1: 多 Agent 系统有哪些常见架构模式？各有什么优缺点？
**要点**：中心化（控制力强但有瓶颈）、去中心化（灵活但协调复杂）、层级式（适合大型任务）、流水线（适合阶段性任务）。

### Q2: 多 Agent 之间如何通信？
**要点**：三种方式 — ① 消息传递（异步、灵活）② 共享状态（同步、简单）③ 对话/辩论（适合决策）。选择取决于协作模式和实时性要求。

### Q3: 如何避免多 Agent 系统中的无限循环？
**要点**：① 设置最大轮次/Token 预算 ② 超时机制 ③ 检测重复消息 ④ Orchestrator 强制终止权 ⑤ 明确任务完成条件。

### Q4: 什么时候该用多 Agent？什么时候单 Agent 就够了？
**要点**：单 Agent 适合简单、同质任务；多 Agent 适合需要多种专业知识、可并行分解、需要交叉验证的复杂任务。注意多 Agent 的额外通信成本。

### Q5: 如何设计多 Agent 系统中的角色？
**要点**：① 职责单一清晰 ② 能力互补不重叠 ③ 工具按角色分配 ④ System Prompt 差异化 ⑤ 明确每个角色的输入输出格式。
