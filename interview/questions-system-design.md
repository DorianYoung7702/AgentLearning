## 题目一：设计一个企业知识库问答 Agent

### 题目描述
为一家中型企业设计一个基于内部知识库的智能问答 Agent，员工可以通过对话方式查询公司制度、产品手册、技术文档等。

### 参考答题思路

**1. 需求分析**
- 用户：公司全体员工（技术和非技术）
- 知识源：公司制度文档（PDF/Word）、产品手册、FAQ、Wiki
- 核心需求：准确回答 + 引用来源 + 权限控制
- 非功能需求：响应 < 5s、99.9% 可用性

**2. 整体架构**
```
用户 → [Web/IM 接口] → [API Gateway]
  → [Query Processor] (查询改写、意图分类)
  → [RAG Pipeline] (混合检索 + Rerank)
  → [LLM Generator] (基于检索结果生成回答)
  → [Guard Rails] (输出安全检查)
  → 返回回答 + 来源引用
```

**3. 核心设计点**
- **RAG 管道**：混合检索（向量+BM25），Rerank 精排
- **Chunk 策略**：按文档结构切分，附加元数据（标题、部门、更新时间）
- **权限控制**：不同部门看到不同文档范围
- **对话记忆**：支持多轮追问
- **兜底策略**：搜不到时明确告知，不编造答案

**4. 技术选型**
- Embedding：text-embedding-3-small 或 bge-large-zh
- 向量数据库：Milvus（规模大）或 Chroma（原型）
- LLM：GPT-4o / Qwen-72B
- 框架：LlamaIndex（RAG 为主）

**5. 评估方案**
- 准确率：人工标注测试集
- Faithfulness：RAGAS 自动评估
- 用户满意度：反馈按钮

## 题目二：设计一个自动化数据分析 Agent

### 题目描述
设计一个能自动分析业务数据的 Agent。用户用自然语言描述分析需求，Agent 自动查询数据库、执行分析、生成图表和报告。

### 参考答题思路

**1. 需求分析**
- 用户：业务分析师、运营人员
- 输入：自然语言分析需求（如"分析上月各产品线的销售趋势"）
- 输出：数据分析结论 + 可视化图表 + 导出报告
- 数据源：MySQL/PostgreSQL 业务数据库

**2. Agent 架构**
```
用户需求
  ↓
[理解模块] → 意图识别 + NL2SQL
  ↓
[规划模块] → 分析步骤规划
  ↓
[执行模块] → 循环执行
  ├── SQL 查询工具
  ├── Python 数据分析工具（pandas）
  ├── 可视化工具（matplotlib/echarts）
  └── 报告生成工具
  ↓
[输出模块] → 结果汇总 + 报告生成
```

**3. 核心设计点**
- **NL2SQL**：自然语言转 SQL，需要提供表结构和字段描述
- **代码沙箱**：数据分析代码在隔离环境执行
- **安全性**：只允许 SELECT 查询，禁止修改数据
- **渐进式展示**：边分析边展示中间结果
- **错误恢复**：SQL 报错时自动修正重试

**4. 工具设计**
```
- query_database(sql): 执行 SQL 查询，返回结果
- run_python(code): 在沙箱中执行 Python 代码
- create_chart(data, chart_type): 生成可视化图表
- generate_report(sections): 生成分析报告
```

**5. 安全考量**
- SQL 注入防护
- 数据脱敏
- 查询超时和结果集大小限制
- 审计日志

## 题目三：设计一个多 Agent 客服系统

### 题目描述
为一个电商平台设计智能客服系统，能处理商品咨询、订单查询、退换货、投诉等多种场景。

### 参考答题思路

**1. 多 Agent 架构**
```
用户消息
  ↓
[Router Agent] → 意图分类
  ├── [FAQ Agent]     → 常见问题（商品信息、规则说明）
  ├── [Order Agent]   → 订单查询、物流追踪
  ├── [Return Agent]  → 退换货处理
  ├── [Complaint Agent] → 投诉处理
  └── [Transfer Agent]  → 人工转接
```

**2. 角色设计**

| Agent | 工具 | 职责 |
|-------|------|------|
| Router | 意图分类模型 | 分流到对应 Agent |
| FAQ | 知识库检索 | 回答常见问题 |
| Order | 订单API、物流API | 查询订单和物流 |
| Return | 退货API、审批流 | 处理退换货流程 |
| Complaint | 工单系统、升级流程 | 处理投诉和升级 |

**3. 核心设计点**
- **上下文传递**：Agent 切换时传递对话历史
- **升级机制**：复杂问题自动转人工
- **情绪检测**：识别用户情绪，优先处理负面情绪
- **知识同步**：所有 Agent 共享最新的商品和政策知识

**4. 性能要求**
- 首响 < 2s
- 90% 问题自动解决
- 人工转接 < 10%

## 题目四：设计一个编程助手 Agent

### 题目描述
设计一个编程助手 Agent，能理解用户的编程需求，生成代码，调试错误，解释代码。

### 参考答题思路

**1. 核心能力**
- 代码生成：根据自然语言需求生成代码
- 代码解释：解释已有代码的功能
- Bug 修复：分析和修复代码错误
- 代码审查：审查代码质量并提出改进建议

**2. Agent 设计**
```
用户需求
  ↓
[意图识别] → 生成/解释/调试/审查
  ↓
[上下文收集]
  ├── 当前文件内容
  ├── 项目结构
  ├── 相关文件（import 依赖）
  └── 错误信息（如有）
  ↓
[LLM 推理] + [代码搜索工具]
  ↓
[代码生成/修改]
  ↓
[验证] → 语法检查 / 测试执行
  ↓
[应用修改] → 编辑文件
```

**3. 工具设计**
```
- read_file(path): 读取代码文件
- edit_file(path, old, new): 编辑代码文件
- search_codebase(query): 搜索代码库
- run_command(cmd): 执行命令（沙箱）
- run_tests(path): 运行测试
```

**4. 关键挑战**
- 理解完整项目上下文（代码库可能很大）
- 代码修改的准确性（不能引入新 Bug）
- 多文件修改的一致性
- 安全执行用户代码

## 题目五：设计一个 Agent 编排平台

### 题目描述
设计一个通用的 Agent 编排平台，让用户通过拖拽的方式构建自定义 Agent 工作流。

### 参考答题思路

**1. 核心概念**
- **节点（Node）**：工作流中的处理单元
  - LLM 节点：调用大模型
  - 工具节点：执行外部工具
  - 条件节点：分支判断
  - 循环节点：重复执行
  - 人工审核节点：Human-in-the-loop
- **边（Edge）**：节点之间的连接和数据流
- **变量（Variable）**：节点之间传递的数据

**2. 系统架构**
```
[前端 - 可视化编辑器]
  ├── 拖拽式画布
  ├── 节点配置面板
  └── 调试和测试面板

[后端 - 执行引擎]
  ├── 工作流解析器
  ├── 节点执行器
  ├── 状态管理
  └── 调度器

[基础设施]
  ├── LLM 网关
  ├── 工具注册中心
  └── 监控和日志
```

**3. 关键设计点**
- **DAG 执行引擎**：支持并行、分支、循环
- **状态持久化**：长流程可暂停和恢复
- **版本管理**：工作流版本控制和回滚
- **模板市场**：预构建的常见工作流模板
- **权限管理**：不同用户访问不同工作流和工具

**4. 参考产品**
- Dify、Coze、n8n、Flowise

## 答题通用框架

面试系统设计题时，建议按以下框架组织回答：

```
1. 需求澄清（2分钟）
   - 用户是谁？规模多大？
   - 核心功能 vs 可选功能
   - 非功能需求（性能、可用性、安全性）

2. 高层架构（5分钟）
   - 画出系统架构图
   - 标注核心组件和数据流

3. 核心组件详设（10分钟）
   - Agent 架构和角色设计
   - 工具选择和设计
   - RAG / 记忆策略
   - 规划范式

4. 技术选型（3分钟）
   - 模型选择
   - 框架选择
   - 基础设施

5. 关键挑战和应对（5分钟）
   - 可靠性
   - 安全性
   - 成本
   - 评估方案
```
